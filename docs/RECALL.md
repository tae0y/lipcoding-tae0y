# RECALL — 입코딩대회 참가 후기

🎙GitHub Copilot 천하제일 입코딩대회에 다녀왔습니다. **Womxn Who Code**가 주관하고 **Jabra·메가존클라우드·Microsoft**가 후원한 대회로, 도시락부터 Copilot Max 구독·헤드셋·컨설팅 지원까지 후원사 덕분에 풍성했어요. 좋은 자리 마련해주신 운영진과 후원사 분들께 감사드립니다. 부슬비가 내리는 광화문... 정말 운치가 있었습니다.

## Code Without Barrier — 장벽이 얼마나 사라졌으면

대회 캐치프레이즈는 "Code Without Barrier"였습니다. 그런데 그 장벽이 얼마나 낮아졌는지, 이번엔 아예 **키보드 없이** 코딩을 하라고 했어요. 타이핑은 물론 마우스나 트랙패드도 쓸 수 없었습니다. 시작 전 1시간과 종료 전 30분만 '키보드 타임'으로 주어지고, 나머지 시간은 전부 보이스챗으로 GitHub Copilot에게 말을 걸어 코딩하는 방식이었습니다.

오프닝 키노트에서 들은 MS 솔루션 엔지니어 이희진님의 한마디가 대회 내내 머릿속에 남았습니다.

> **"보이스코딩은 타이핑이 금지되는 것이지, 엔지니어링이 금지되는 것이 아니다."**

도구는 극단적으로 편해졌지만, 좋은 소프트웨어를 만들기 위한 제약·검증·피드백의 설계는 오히려 더 중요해진다는 이야기였습니다. 실제로 키보드 없이, 사람의 개입 없이 반나절을 집중해 코딩해보니, 도리어 사람이 어느 지점에서 개입해야 하는지가 더 또렷하게 느껴졌습니다.

## 떠올림 — 쏟아지는 아이디어를 붙잡는 받은편지함

저는 "떠올림"이란 할일 관리 앱을 만들었습니다. 평소 로컬 에이전트와 텔레그램으로 쓰던 기능을 앱으로 옮겨봤어요.

AI 어시스턴트의 발전으로 정보와 산출물의 양은 폭발적으로 늘었지만, 사람의 뇌가 한 번에 처리할 수 있는 용량은 변하지 않았습니다.

떠올림은 이 간극을 메우는 도구입니다. 아이디어가 떠오른 순간 앱에 던져두면, AI가 "지금 바로 시작할 수 있는가"를 판단하고 정보가 부족한 항목은 백로그로 분류합니다. 그리고 어느 날 충분히 여유가 있는 날에, 미리 조사해둔 맥락 자료와 후속 액션 아이템을 정리해 제안합니다. 쏟아지는 아이디어를 놓치지 않고 붙잡되, 당장은 중요한 일에 할 수 있는 만큼 집중하자는 컨셉의 앱입니다.

사용한 기술은 다음과 같습니다.

| 레이어 | 사용한 기술 | 비고 |
| --- | --- | --- |
| **프론트** | React 19 + Vite 6 + Tailwind 4 (TS) | - |
| **백엔드** | FastAPI + Pydantic 2 + SSE | Pydantic으로 출력 스키마 정의. |
| **AI** | Copilot SDK + openai(Azure) | Azure AI Foundry 모델 서빙, BaseURL 지정하여 연결. |
| **데이터** | SQLite + Azure Files 볼륨 | 볼륨 마운트로 재시작 후 영속성 확보. |
| **배포** | Azure Container Apps | AI는 App Service를 추천했지만 반복 실패 후 Container Apps로 선회. |

* 좋았던 선택
  * **openapi.json으로 스펙을 먼저 정의하고 Spectral CLI로 검증:** 그 스펙을 기준으로 프론트와 백엔드를 독립적으로 구현했습니다. 인터페이스가 고정되어 있으니 병렬 작업이 충돌 없이 맞물렸습니다.
  * **Playwright와 Chrome DevTools로 테스트:** 보이스코딩 환경에서도 동작 확인의 신뢰도를 유지할 수 있었습니다.

* 아쉬웠던 선택
  * **Azure 배포 스켈레톤을 미리 준비하지 않은 것:** `Container Apps + ACR + Azure Files 볼륨` 조합을 대회 중에 처음부터 구성하느라 시간이 많이 들었습니다. 배포 환경만큼은 동작하는 뼈대를 사전에 갖춰두는 게 맞았습니다.
  * **AI auto 모드에서 중간 체크를 생략한 것:** 긴 작업을 통으로 맡기고 결과만 받았는데, 중간에 한 번 개입해 방향을 확인했다면 범위 축소를 더 일찍 잡을 수 있었을 겁니다.

## GitHub Copilot과 Copilot SDK의 놀라운 발전

가장 먼저 놀란 건 Copilot 제품 자체의 변화였습니다. chatmode나 prompts를 얹어 쓰던 시절과는 꽤 달라졌습니다. 서브 에이전트 오케스트레이션 기능도 강력했고, VSCode에서 여러 채팅 세션을 한 번에 볼 수도 있습니다. skills, hooks 익숙한 기능들도 보이고, instructions와 prompts 등 Copilot만의 기능도 새삼 새로웠습니다.

Copilot SDK의 장점은 이런 기능들을 내 앱으로 직접 가져올 수 있다는 점입니다. 「떠올림」의 자동 사전조사 기능에 붙여봤고, 세션 + 도구 호출 + SSE 스트리밍을 하나의 흐름으로 엮었습니다. 실제 구현에서 모델은 두 개의 도구를 순서대로 호출합니다. search_web으로 자료를 수집하고, collect_materials로 결과를 store에 저장하는 구조였습니다.

대회가 끝나고 회고글을 쓰면서 코드를 다시 들여다봤을 때는 아쉬움이 남았습니다. "에이전트 오케스트레이션을 제대로 썼다"고 생각했는데 사실은 그렇지 않았습니다. 실제 구조는 단일 세션에서 두 도구를 순서대로 호출하는 것에 가까웠습니다. 도구는 외부 검색을 하는 것이 아니라 모델의 출력을 받아 적었고, 조사와 제안 작업을 검증 루프 없이 단일 호출에 도구 형식을 입힌 쪽이었습니다.

원래 의도는 달랐습니다. Copilot의 서브 에이전트 오케스트레이션을 활용해 웹검색 에이전트가 실제로 외부를 조회하고, 그 결과를 다음 턴으로 넘기는 진짜 멀티턴 루프를 만들려 했습니다. 그러나 요구사항을 충분히 정의해두지 않은 탓에, 구현 결과는 가장 단순한 형태로 끝나버렸습니다. SDK가 에이전틱 기능을 지원한다는 것과, 내가 실제로 그 깊이까지 활용했는가는 전혀 다른 문제였습니다.

## ???: 커밋해줘. 아니 LG 컴퓨터 말고. 코밑 말고, 커뮤니티도 아니고, 커-밋.

보이스코딩에서 가장 먼저 걸리는 장벽은 한국어 인식이었습니다.

GitHub Copilot은 보이스챗 기능을 제공하는데, 한국어 언어팩을 설치하고 `settings.json`에 `"accessibility.voice.speechLanguage": "ko-KR"` 설정을 추가하면 사용할 수 있습니다. 다만 인식률이 바이브코딩에 바로 쓸 만큼 만족스럽지 않아 대체재를 찾았습니다.

대회 당일 사전세팅에서 이 문제를 발견했고, 유료 옵션 대신 무료로 쓸 수 있는 **VoiceInk**를 설치해 `large-v3-turbo` 모델을 받아 사용했습니다. 앱 단축키 설정은 이미 키보드 타임이 끝난 뒤라 Copilot에게 부탁해 macOS 내장 받아쓰기를 VoiceInk로 교체했습니다.

그래도 프로그래밍 용어는 여전히 헷갈렸습니다. "커밋"이 "컴퓨터"나 "커뮤니티"로 받아쓰기 되는 식이에요. 영어 기술 용어를 한국어 발음으로 말하는 구간이 STT의 약점입니다. 그래서 `AGENTS.md`에 보이스 입력 전용 섹션을 추가해뒀습니다. 전사 오류가 있어도 맥락으로 추론해 진행하고, 가역적인 단계는 확인 없이 처리하되 파괴적인 결정만 멈추고 확인하는 구조입니다. 음성 입력이라는 걸 AI가 인지하면, 잘못 받아쓰인 단어도 적절히 해석해서 처리해줬습니다.

```markdown
## Voice-dictated instructions

My instructions reach you through a microphone (speech-to-text), so the transcript may contain
errors, missing words, or odd phrasing.

- Infer my intent from context and proceed — don't stall on small transcription glitches.
- Don't ask for confirmation on routine, reversible steps; just do them.
- Only stop to confirm when something is genuinely necessary and ambiguous — e.g. a destructive
  or hard-to-reverse action, or a decision that materially shapes the app.
- If a specific word or term is unclear or seems mis-transcribed, ask what I meant rather than guessing.
```

대회장에서 특이한 일도 있었습니다. 마이크로소프트 광화문 건물 네트워크가 일본 VPN을 경유했는지, Copilot 응답에 일본어·영어·한국어가 뒤섞여 나왔습니다. 처음엔 모델 문제인가 했는데, 네트워크 경로가 원인이었습니다.

## "다 알아서 해줘"는 통하지 않는다

키노트에서 들은 숫자 하나가 인상적이었습니다. 단계당 99% 정확도라도 50단계를 거치면 결과는 50% 수준까지 떨어진다는 이야기였습니다. AI가 갈 수 있는 길이 워낙 넓다 보니, 원하는 방향으로 끌고 가려면 결정론적 통과 기준(명세)이 먼저 있어야 한다는 거죠.

이건 실전에서 그대로 체감됐습니다. PRD, 유저 스토리, openapi.json, 테스트 케이스 시나리오처럼 명확한 명세가 앞에 버티고 있어야 작업이 진행되었습니다. 명세 없이 "알아서 잘 해줘"로 던지면, 그럴듯하긴 해도 매번 다른 결과가 돌아옵니다. 우승팀들의 팁도 결이 같았습니다. 심사 기준을 골든 룰로 만들어 프롬프트에 반영하고, 기준별 Judge 에이전트로 로컬에서 미리 평가한 뒤 제출하는 방식이었습니다. 통과 기준을 먼저 코드 바깥에 고정해두는 거죠.

저도 시간 압박 속에서 이 교훈을 거꾸로 배웠습니다. 1차 심사 피드백을 던지며 "남은 17분, 가능한 걸 진행해줘"라고 맡겼더니, AI가 임팩트 대비 공수가 가장 적은 항목(prompt-injection 가드 한두 줄)을 먼저 집지 못하고, "핵심 PRD 흐름"이라는 큰 작업에만 매달렸습니다. 위임할 때 "공수(분) × 임팩트로 정렬해 top-N만"라고 이과식으로 강조해줘야 했을까요? 모호하게 맡기니 AI가 작업의 우선순위를 파악하지 못했습니다.

## AI는 내가 가본 곳까지만 데려다준다

AI는 결국 사람이 한 번 도달해본 곳까지 우리를 다시 데려다줍니다. 반대로 말하면, 내가 한 번도 걸어보지 않은 길은 AI와 함께라도 훨씬 더디게 가게 됩니다.

이 사실이 대회 내내 여러 층위에서 반복됐습니다.

사전 준비가 먼저였습니다. 대회 전에 openapi.json을 먼저 정의하고 Spectral CLI로 스펙을 검증해 둔 뒤, 프론트와 백엔드를 해당 스펙 기준으로 병렬 구현했습니다. 각각 독립 테스트 후 통합테스트까지 진행하는 플로우도 미리 그려뒀습니다. PRD 작성 단계에서는 요구사항을 집요하게 구체화했고, MS Learn·Context7·Tavily 같은 MCP 서버도 대회 전에 세팅해 뒀습니다. awesome-github-copilot 레포를 통째로 클론해 두고 프론트·백엔드·클라우드 배포에 필요한 스킬 파일을 미리 뽑아두기도 했고요. 기술 스택별로 사전 리서치를 돌린 다음 맥락 문서를 만들어 두고 구현을 시작했습니다. 심지어 대회 진행 시간에 맞춰 셸로 현재 시각을 확인하면서 시간 배분 계획을 Copilot에게 가이드받는 스크립트도 준비했습니다. 이런 준비들이 쌓여 있으니, 보이스 한 마디로 원하는 방향이 빠르게 잡혔습니다.

배포도 마찬가지였습니다. AI는 App Service를 추천했지만 반복 실패 후 처음 가려던 Container Apps로 돌아왔고, 원격 빌드가 느리자 로컬 빌드 후 ACR 푸시로 바꿨습니다. FastAPI가 빌드된 SPA를 같은 origin에서 서빙해 CORS를 없애고, SQLite는 Azure Files 볼륨에 마운트해 영속성을 확보했습니다. 이 판단들은 AI가 제안한 게 아니라 제가 이미 알고 있던 선택이었습니다. AI를 잘 쓰려면, 내가 먼저 그 길을 걸어봤거나 도구를 미리 세팅해 두어야 하는구나 싶었습니다.

## 보안은 생각하지도 못했다

다른 부분은 AI가 정말 잘 해줬는데, 마지막에 심사 에이전트가 결과물을 평가할 때 보안에서 많은 지적을 했습니다. Responsible AI·Security·Trust 항목이 7개 기준 중 유일하게 3점이었습니다.

지적은 날카로웠습니다. 공개 배포된 앱에 인증·권한 분리가 없어 누구나 아이디어 조회·삭제 API를 호출할 수 있고, Key Vault나 managed identity 없이 키 기반 비밀값에 의존하며, 삭제 같은 파괴적 동작에 확인 절차가 없다는 점이었습니다. 프롬프트 인젝션 방어도 명시적이라기보다 도구 스키마에 의존하고 있었고요.

더 뼈아픈 건 우선순위였습니다. 프롬프트 인젝션 방어는 시스템 프롬프트 가드 한두 줄이면 끝나는 저공수·고가시성 작업이었고, 심지어 관련 자료를 미리 정리해 둔 것도 있었습니다. 그런데 막판에 피드백을 받고 빠르게 수정할 때, 그 쉬운 걸 놓쳤습니다. AI에게 "가능한 걸 해줘"라고 던졌더니, 이번에도 점수 효율이 가장 높은 한두 줄이 아니라 가장 묵직해 보이는 작업을 집어 들었어요.

생각해보면 보안은 AI가 알아서 챙겨주는 영역이 아닙니다. 기능은 "동작하면 된다"는 신호가 명확해서 AI가 잘 구현하지만, 보안은 요구하지 않으면 비어 있는 채로 통과해 버립니다. 인증·키 관리·삭제 확인 같은 건 의식적으로 체크리스트에 올려놓고 챙겨야겠다는 숙제가 남았습니다.

## 구현이 쉬워질수록 기획이 무거워진다

마지막은 기획에 대한 반성입니다. GitHub Copilot MAX 구독의 월 과금에 해당하는 크레딧의 50%를 이 하루에 소진했습니다. 그만큼 막대한 AI 리소스를 써서 동작하는 앱을 만들어냈는데, 정작 "이 앱의 ROI는 뭐냐, 고객을 붙잡는 락인 요소가 뭐냐"고 물으면 답할 수 없었습니다.

심사 피드백도 같은 지점을 짚었습니다. 제품 흐름은 설득력 있지만, 절약 시간·감소 단계·오류 예방률 같은 정량 근거가 없어 생산성 효과가 충분히 입증되지 않았다고요. 주간 추천도 실제 스케줄러나 임베딩 랭킹이 아니라 수동 엔드포인트와 단순 우선순위에 머물러 있었고요.

더 근본적인 건, 원래 의도가 구현 과정에서 나도 모르게 축소되었는데 이를 자각하지 못했다는 점입니다. 「떠올림」의 출발점은 '사람의 역량을 보존하기 위해 결정이 아니라 선택지를 제공한다'였습니다. 그런데 막상 만들어진 건 'AI 제안을 수락할지 말지' 정도로 범위가 좁혀져 있었습니다. 선택지를 펼친다는 가치가 수락/거절 버튼으로 축소됐는데도, 만드는 동안엔 알아채지 못했습니다. 앞에선 같은 편향이 '우선순위'에서 드러났다면, 여기선 '제품의 야심'에서 드러난 겁니다. 사람이 기준을 또렷이 붙잡고 있지 않으면, AI는 위임의 빈틈을 늘 가장 단순한 쪽으로 메웁니다.

만드는 일(구현)은 이제 정말 쉬워졌습니다. 보이스 챗만으로도, 충분히 동작하는 앱이 나옵니다. 그래서 오히려 "무엇을, 왜 만드는가" 라는 기획의 날카로움이 차별점이 됩니다. "사람이 하는 것 중 가장 중요한 건 문제 정의 — 우선순위·리스크 판단·최종 책임"이라는 말이 마지막에 다시 떠올랐습니다. 구현이 딸깍으로 끝나는 시대일수록, 결국 문제를 어떻게 정의하느냐에 달려 있는 것 같습니다.

## 마무리

키보드에서 손을 떼고 AI에게 모든 일을 맡기자, 도리어 사람이 어디에 개입해야 할지가 분명해졌습니다. 아이디어를 구체화해 요구사항을 미리 정의해두고, 필요한 도구들도 준비하고, 명세도 준비해놓고, 나름대로 '결정론적 통과기준'을 만든다고는 만들었는데, 정작 아이디어에서 요구사항으로 넘어갈 때, 요구사항에서 명세로 넘어갈 때 누락된 것들이 정말 많았습니다. 그리고 '보안'이라는 영역은 애초에 고려도 하지 못했습니다.

AI 시대는 0→1이 아니라 1→10이라는 회고를 많이 들었고, 저 역시 같은 것을 느꼈습니다. 제가 이미 경험해본 것들을 AI가 더 잘해준다는 것이었습니다. AI의 발전 속도는 정말 눈이 부시지만, 결국 주니어 개발자가 해야 할 일은 전과 다르지 않은 것 같습니다. 꾸준히 견문을 넓히고 경험을 쌓아가야겠습니다.
