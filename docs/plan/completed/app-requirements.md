# 앱 요구사항 명세 (핸드오프)

> 그릴링 세션(2026-06-20)으로 확정한 요구사항 정리. 이후 작업은 이 문서를
> 출발점으로 인계한다. 확정 항목은 그대로 구현하고, "미결정" 항목은 구현 전
> 결정이 필요하다.

## 1. 한 줄 정의

사용자가 아이디어를 던지면 AI가 **"지금 착수 가능한가"**를 판정한다. 가능하면
인박스에 노출하고, 아니면 **덤프**에 보관한다. 정보가 부족한 덤프 항목은 AI가
대신 사전조사를 해둔다. 그리고 **주 1회 퇴근 후 시점**에, 사용자의 부하가 낮을
때 덤프에서 한 건을 **"지금 해볼까요?"**로 제안한다. 계획·판단·통합·실행은
사람이 유지하여 작업기억 부하를 줄인다.

대상 컨텍스트: **퇴근 후 개인 시간**에 해볼 거리를 추천하는 앱. 일과 중 업무
부하는 고려 대상이 아니다.

## 2. 핵심 원칙 (불변)

- **사람 대체 금지 / 영향력 강화(augment, not replace).** AI는 조사·준비 등
  바닥 깔기만 한다.
- **"강화"의 정의:** AI는 *사람이 꼭 해야 하는 일(계획·판단·통합·실행)*은 절대
  대체하지 않는다. 그 주변의 준비·조사만 대신한다.
- 위험/결정 지점은 항상 사람이 최종 승인한다.

## 3. 두 개의 판단

앱에는 서로 다른 두 판단이 존재한다. 섞이지 않도록 분리해서 구현한다.

### 3.1 입구 판정 (아이디어 캡처 시)

아이디어가 들어오면 **2요소**로 판정한다.

1. **정보 준비성** — "구체적인 다음 액션(next action)이 정확히 특정 가능한가?"
2. **현재 여유** — "현재 일정/할일과 충돌하지 않는가? + 사용자 감정 상태 OK인가?"

판정 결과:

- 둘 다 OK → `status: inbox` ("지금 착수 가능")으로 노출
- 하나라도 NO → `status: dump` (보류) + **사유 태그** 부여

덤프 사유 태그:

| 태그 | 의미 | AI 동작 |
|------|------|---------|
| `info_gap` | 정보가 부족해서 보류 | **AI 사전조사 실행** (빈틈 채우기) |
| `no_capacity` | 정보는 충분하나 지금 여유가 없어 보류 | 조사 스킵, 주간 트리거 대기 |

### 3.2 제안 트리거 (주 1회, 퇴근 후)

- **발동 방식:** 스케줄 배치 push (실시간 모니터링 아님). 사용자가 설정한
  **요일 + 시각**에 잡이 1회 실행된다.
- **시각:** 주 1회, 사용자 설정 요일·시각 = **퇴근 시각 + 20~30분 버퍼**
  (퇴근 후 숨 돌리는 시간).
- **부하 게이트:** 그 시점에 퇴근 후 가용성(저녁 약속/할일 + 감정)을 확인하여
  저녁이 비고 감정이 받쳐줄 때만 추천한다. 아니면 그 주는 침묵/연기.
- **추천 랭킹 기준 3개:**
  1. 사용자 부하가 낮음 (캘린더 + 투두 + 감정)
  2. 사전조사가 충분히 되어 있음
  3. (항목이 많을 때) 현재 사용자 관심사·진행 중인 일과의 **연관성**
- **연관성 신호 출처:** 기존 데이터 재활용 — 인박스에 살아있는 항목 + 투두 +
  최근 캡처한 아이디어로 "관심 프로파일"을 구성하고, Azure OpenAI 임베딩
  유사도로 덤프 항목과 매칭한다. *추가 입력·외부 인증 없음.*

## 4. AI 사전조사 산출물

`info_gap` 덤프 항목에만 실행한다. 산출물 범위는 **재료 + 선택지/프레임 정리**
까지로 제한한다.

- **재료:** 빠진 정보·참고 사실·링크 등.
- **선택지/프레임:** "이걸 하려면 보통 A/B/C 옵션이 있다" 수준의 구조화.
- **최종 "다음 액션 결정"은 사람의 몫**으로 남긴다. (추천 액션 초안까지 제시하는
  것은 대체 위험이라 배제.)

## 5. 제안 UI

주간 추천이 떴을 때:

- 추천 1건 제시 + **추천 근거 3줄**(부하 낮음 / 사전조사 완료 / 관심사 연관성).
- 결정 버튼은 사람: `[지금 해볼게요 / 다음에 / 관심없음]`.
- 근거 노출로 투명성을 확보하고 사용자가 납득하여 주도하게 한다.

## 6. 화면 구성 (2화면으로 단순화)

기존 3영역(인박스/덤프/제안)을 **입력 화면 / 조회 화면** 2개로 합친다.

### 화면 1 — 입력

- 아이디어 한 줄 입력(주 기능).
- "오늘 상태" 입력: 감정 3버튼(나쁨/보통/좋음), 투두 입력, 캘린더 토글, 주간
  트리거 요일·시각 설정.
- 입력 즉시 AI 판정 → "착수 가능" 또는 "덤프(사유)" 결과를 짧게 피드백.

### 화면 2 — 조회

하나의 리스트를 섹션으로 구분한다.

- **이번 주 추천** — 트리거 결과 카드 + 근거 3줄 + 승인 버튼 (맨 위).
- **지금 착수 가능** — `inbox` 항목.
- **보관함/덤프** — 사유 태그 + 사전조사 노트(펼치기).

## 7. 데이터 모델 (초안)

```text
Idea {
  id
  text                      // 원문 아이디어
  status: inbox | dump      // inbox=착수가능, dump=보류
  dumpReason: info_gap | no_capacity | null
  research: {               // info_gap일 때만 AI가 채움
    materials: [...]        // 재료(사실·링크)
    options:   [...]        // 선택지/프레임 (결정은 사람)
  } | null
  createdAt
}

UserState {                 // 부하 신호
  emotion: bad | normal | good
  todos: [...]              // 직접 입력 or 연동
  calendarBusy: bool        // 저녁 가용성
  triggerSchedule: { weekday, time }   // 주 1회 (퇴근 + 버퍼)
}

Suggestion {                // 주간 트리거 결과
  ideaId
  reasons: { lowLoad, researchDone, relevance }  // 근거 3줄
  decision: accepted | postponed | dismissed | null
}
```

## 8. Copilot SDK 활용 지점 (심사 25%)

- **입구 판정 도구:** 다음 액션 특정 가능 여부 판단 + 부족 정보 도출.
- **사전조사 도구:** `info_gap` 항목의 빈틈을 조사해 재료/선택지 생성
  (조사 → 구조화 도구 체인).
- **연관성 매칭:** Azure OpenAI 임베딩으로 관심 프로파일 ↔ 덤프 항목 유사도.
- 스트리밍 출력으로 조사/판정 과정을 보여준다.

## 9. 미결정 항목 (구현 전 결정 필요)

- [x] **조회 화면 레이아웃:** **확정 — 섹션 한 스크롤**(추천→인박스→덤프, 탭 없음).
      ASCII 와이어프레임은 [ui-wireframes.md](./ui-wireframes.md).
- [x] **캘린더 연동 방식:** **확정 — 외부 연동 없음.** 저녁 가용성 토글 1개
      (`calendarBusy`)만 사용자가 직접 입력.
- [x] **투두 출처:** **확정 — 앱 내 한 줄 수동 입력.** 외부 연동 없음.
- [x] **부하 게이트 임계치:** **확정 — `calendarBusy=false` AND `emotion≠bad`**
      일 때만 추천. boolean 2개 조합으로 명확, 별도 튜닝 없음.
- [x] **`no_capacity` 항목의 재평가 시점:** **확정 — 매 주간 트리거 때 재평가.**
      입구 판정 재실행 안 함. 트리거 잡이 그 시점 상태로 다시 본다.
- [x] **추천 건수:** **확정 — 주간 트리거 시 1건만.** 작업기억 부하 최소화 원칙.
- [x] **연관성 기준(3번) 적용 조건:** **확정 — 덤프 항목 ≥ 5개일 때만** 임베딩
      유사도 랭킹 적용. 미만이면 사전조사 완료도 순.
- [x] **사전조사 트리거 타이밍:** **확정 — 덤프 진입 즉시**(동기 호출 + 스트리밍
      노출). SDK 깊이(심사 25%) 데모에 유리, 단일 사용자라 비용 부담 적음.
- [x] **데이터 영속화:** **확정 — SQLite 파일 1개 (`/data/app.db`)를 Azure Files
      볼륨에 마운트.** 임베딩 벡터는 JSON/TEXT 컬럼에 저장, 파이썬에서 코사인
      유사도 brute-force 계산(항목 수 적음). 매니지드 DB·벡터 DB·Redis 안 씀.
- [x] **인증:** **확정 — 단일 사용자 앱, 로그인/회원가입 없음.** 규칙상 인증은
      필수요소 아님(필수: 웹앱/Copilot SDK/Azure). 대신 심사 #6(6%) 대비 secret
      (Azure OpenAI 키 등)은 환경변수로만 처리.
- [x] **감정 입력 주기:** **확정 — 매 아이디어 입력 시 3버튼(나쁨/보통/좋음),
      마지막 값 유지.** 별도 입력 단계 없음. (외부 연동 전면 배제: 캘린더·투두·
      감정 모두 앱 내 수동 입력)

## 11. 기술 스택 & 배포 (확정)

| 계층 | 선택 | 비고 |
|------|------|------|
| 프론트엔드 | React + Vite + Tailwind | 화면 2개, 기본 컴포넌트 위주 |
| 백엔드 | FastAPI (Python) | OpenAPI 스펙은 `docs/plan/openapi.yaml` |
| AI 모델 | Azure OpenAI (판정·조사·임베딩) | 키는 환경변수 |
| 배포 | **Azure Container Apps (ACA)** | 컨테이너 이미지로 배포 |
| 저장소 | **SQLite `/data/app.db` on Azure Files 볼륨** | 컨테이너 휘발성 회피, replica 1개 고정 |

- **ACA 스케일:** `minReplicas=1, maxReplicas=1` — SQLite 다중 쓰기 회피 +
  콜드스타트 제거(단일 사용자라 충분).
- **폴백:** Azure Files 마운트가 사전 준비 때 안 뚫리면 이미지 내장 SQLite로
  운영하되 "데모 한 세션 내 재시작 금지"로 데이터 생존 포기.

## 12. 스코프 가드 (하지 말 것)

- 화면 2개 초과로 확장 금지.
- 인증/회원가입 풀구현 금지(단일 사용자 가정).
- 실시간 백그라운드 부하 모니터링 금지(주 1회 배치로 충분).
- AI가 "다음 액션"을 사람 대신 확정하는 것 금지(강화 원칙 위반).
