# 천하제일 입코딩 대회 2026 — 시간 배분 계획

## 0. 게임의 규칙 (가장 먼저 기억할 것)

| 구분 | 시간 | 비고 |
|------|------|------|
| 사전 준비 | 10:30 - 11:30 (1시간) | **키보드 허용** — 여기서 최대한 깔아두기 |
| 점심 | 11:30 - 12:30 | 머릿속으로 흐름 리허설 |
| 입코딩 | 12:30 - 16:30 (4시간) | **음성만 허용** — 실제 개발 |
| 심사·발표 | 16:30 - 17:30 | 제출 마감 16:30 엄수 |

**필수 3종 (못 지키면 탈락 리스크):**

1. 반드시 **웹 앱**
2. 반드시 **Copilot SDK** 사용
3. 반드시 **Azure 배포**

## 1. 심사 가중치 → 시간 투자 우선순위

| 순위 | 항목 | 가중치 | 전략적 함의 |
|---|------|---:|------|
| 1 | Effective Use of Copilot SDK | 25% | 앱의 **핵심 가치**가 SDK에서 나와야. 깊이 > 기능 수 |
| 2 | Productivity Impact & Problem Fit | 18% | 작업기억 부하 감소라는 명확한 문제·대상 강조 |
| 2 | Azure AI & Cloud Integration | 18% | 모델 계층을 **Azure OpenAI / Foundry** 위에서 |
| 4 | Functionality & Technical Execution | 16% | 엔드투엔드 동작이 최소 조건 |
| 5 | User Experience & Workflow Design | 12% | 저항 낮은 UI, 우아한 로딩/에러 처리 |
| 6 | Responsible AI, Security & Trust | 6% | 위험 작업 전 사람 확인, secret 안전 처리 |
| 7 | Innovation & Originality | 5% | 아이디어(작업기억 보존)가 이미 차별점 |

> **결론:** SDK(25) + Azure AI(18) = 43%. 이 둘에 가장 많은 시간을 쓰되,
> "엔드투엔드로 돈다(16%)"는 바닥을 절대 놓치지 말 것.

## 2. 제품 한 줄 정의 (스코프 고정)

> 사용자가 아이디어를 던지면 AI가 **"지금 착수 가능한가"**를 판단해
> 가능하면 표시하고, 아니면 **덤프**에 보관한다. 덤프 항목은 AI가 대신
> 사전 조사를 해두고, 사용자의 부하가 낮을 때 **"지금 해볼까요?"**로 제안한다.

핵심 화면은 **3개**로만 고정한다. 욕심내면 4시간 안에 못 끝낸다.

1. **인박스/캡처** — 아이디어 입력 → AI가 착수 가능 여부 판정
2. **덤프 리스트** — 보관된 항목 + AI가 채워둔 사전 조사 노트
3. **제안 패널** — "지금 해볼 만한 일" 추천 (사람이 최종 결정)

## 3. 단계별 시간 배분

### 사전 준비 (10:30-11:30, 키보드 OK) — 인프라/스캐폴딩 끝내기

음성 코딩은 느리다. **타이핑 가능한 이 1시간에 "보일러플레이트"를 최대한 끝낸다.**

- [ ] 리포 + 프로젝트 스캐폴딩 (프론트 + 백엔드 골격)
- [ ] Copilot SDK 의존성 설치 + "hello world" 한 번 호출 성공 확인
- [ ] Azure 리소스 미리 생성: **Azure OpenAI 배포(모델)** + 호스팅(App Service/Container Apps/Static Web Apps)
- [ ] `.env` / secret 주입 경로 확인 (배포 시 secret 처리)
- [ ] 배포 파이프라인 1회 성공 (빈 앱이라도 **배포가 된다**는 걸 검증)
- [ ] 데이터 모델 스텁: `Idea { id, text, status: inbox|dump|suggested, research }`

> 사전 준비의 목표는 "기능"이 아니라 **"막힐 만한 곳을 다 뚫어두기"**.
> 특히 Azure 배포와 SDK 첫 호출은 반드시 여기서 성공시켜 둔다.

### 점심 (11:30-12:30) — 무음 리허설

- 음성으로 부를 명령/프롬프트 문장을 머리/메모로 정리
- "컴퓨터→커뮤니티" 같은 오인식 대비: 짧고 명확한 발화 문장 준비
- 4시간 동안의 작업 순서를 1~2~3으로 외우기

### 입코딩 4시간 (12:30-16:30, 음성만)

| 블록 | 시간 | 목표 | 산출물 |
|------|------|------|--------|
| A | 12:30-13:15 (45분) | 데이터 + API 골격, 아이디어 CRUD | 인박스에 항목 추가/조회 동작 |
| B | 13:15-14:15 (60분) | **Copilot SDK 핵심 로직** | "착수 가능 판정" + 덤프 분류 (도구 호출/프롬프트 설계) |
| C | 14:15-15:00 (45분) | **덤프 자동 사전 조사** + 제안 | SDK가 덤프 항목 리서치 노트 생성, 제안 패널 |
| D | 15:00-15:40 (40분) | UX 마감 | 스트리밍 출력, 로딩/에러 처리, 위험 작업 전 확인 |
| E | 15:40-16:10 (30분) | **배포 + 엔드투엔드 검증** | 실제 Azure URL에서 전 흐름 동작 |
| F | 16:10-16:30 (20분) | 제출 + 데모 준비 | 제출 폼 작성, 데모 시나리오 1개 |

### 버퍼 원칙

- 각 블록은 **"동작하는 것"**을 우선 — 안 되면 다음 블록으로 넘어가고 나중에 복귀
- **15:40 배포는 마감이 아니라 데드라인.** 그 전에 한 번이라도 배포되어 있어야 함
  (사전 준비에서 이미 1회 배포했으니, 여기선 "재배포"라 안전)

## 4. 점수를 끌어올리는 결정타 (시간 남으면 / 우선 챙기면)

- **SDK 깊이 (25%):** 단순 한 번 호출이 아니라 **에이전트 + 도구 호출 + 스트리밍**을
  보여주기. "착수 가능 판정"과 "사전 조사"를 각각 도구(tool)로 설계하면 깊이 어필.
- **Azure (18%):** 모델은 반드시 **Azure OpenAI** 경유. 데모에서 "Foundry/Azure OpenAI 위에서 돈다"를 명시.
- **Responsible AI (6%):** "지금 해볼까요?" 제안을 **사람이 승인**하는 구조 자체가 가점 포인트 — 명시적으로 보여줄 것.
- **Innovation (5%):** "작업기억 5개 한계 보존"이라는 인지과학 프레이밍을 데모 멘트에 넣기.

## 5. 절대 하지 말 것 (시간 도둑)

- 인증/회원가입 풀구현 (단일 사용자 가정, 로컬/임시로)
- 화려한 디자인 시스템 (기본 컴포넌트 + Tailwind 정도)
- 화면 4개 이상으로 확장
- 음성으로 대규모 리팩터링 시도 (오인식·시간 낭비)
- 사전 준비 시간에 "기능" 만들기 (그 시간은 인프라 뚫는 데만)
