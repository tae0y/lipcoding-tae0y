# FEEDBACK — 입코딩대회 채점 결과

**종합 평가**

「떠올림 (Working Memory Inbox)」은 사용자의 작업기억을 보호한다는 문제의식이 뚜렷하고, 이를 한 줄 캡처, AI 판정, 정보 부족 아이디어의 사전조사, 낮은 부담의 주간 추천으로 이어지는 설득력 있는 제품 흐름으로 구현한 점이 강점입니다. 단순한 채팅 래퍼가 아니라 `CopilotClient.create_session`과 도구 호출을 활용해 아이디어의 준비도와 사용자의 현재 여유를 판단하고, 사전조사 결과를 구조화된 데이터로 조립하는 방식은 Copilot SDK의 역할을 제품 핵심 가치와 잘 연결했습니다. FastAPI, SQLite, Azure OpenAI, Azure Container Apps 기반의 실제 배포 흐름도 잘 갖춰져 있으며, AI 실패 시 폴백을 둔 점은 기술 실행 측면에서 안정감을 줍니다. 사용자 경험도 전반적으로 매끄럽습니다. 한국어 마이크로카피와 진행 표시, 스켈레톤, 추천 근거, 수락/연기/제외 선택지는 사용자가 AI에게 결정을 넘기는 것이 아니라 부담을 줄이면서 최종 판단을 유지하도록 돕습니다. 특히 “정보가 부족한 아이디어만 조용히 조사하고, 부하가 낮을 때 하나만 제안한다”는 설계는 생산성 문제에 잘 맞는 독창적인 접근입니다. 다만 일부 핵심 약속은 아직 완성도가 제한적입니다. 주간 추천은 실제 스케줄 배치나 임베딩 기반 랭킹이라기보다 수동 엔드포인트와 단순 우선순위에 가깝고, 도구 호출도 외부 검색이나 실제 데이터 조회보다는 모델 출력을 구조화하는 채널로 활용되는 비중이 큽니다. 생산성 효과 역시 절약 시간, 단계 감소, 오류 예방 같은 정량 근거가 부족해 임팩트가 충분히 입증되지는 않았습니다. 또한 공개 배포된 앱에서 인증과 권한 분리가 없고, Key Vault나 managed identity 없이 키 기반 비밀값에 의존하며, 삭제 확인·작업 취소·결정 되돌리기 같은 사용자 통제 장치가 제한적인 점은 신뢰와 안전 측면의 주요 약점입니다. 가장 영향력 있는 개선은 “실제 운영 가능한 개인 비서”로서의 신뢰성과 자동화를 강화하는 것입니다. 인증/권한, 안전한 비밀 관리, 삭제 확인과 되돌리기 같은 기본 보호 장치를 먼저 보강하고, 동시에 주간 추천을 실제 스케줄러와 임베딩 기반 관련도 평가로 연결하면 제품의 핵심 약속인 “작업기억을 덜어 주는 자동 인박스”가 훨씬 더 설득력 있게 완성될 것입니다.

**Effective Use of Copilot SDK4/5**

Copilot SDK 사용은 앱의 핵심 가치인 ‘아이디어 분류’와 ‘info_gap 사전조사’에 직접 연결되어 있어 강한 수준입니다. 단순 채팅 래퍼가 아니라 `CopilotClient.create_session`에 도구를 붙여 정보 준비성/현재 여유를 판정하고, 사전조사에서는 구조화된 도구 호출 결과를 최종 데이터 모델로 조립하며 SSE 스트리밍까지 제공합니다. 다만 도구가 외부 검색이나 실제 데이터 조회를 수행하기보다는 모델 출력의 구조화 채널로 주로 쓰이고, 주간 추천의 임베딩 유사도/고급 컨텍스트 관리는 아직 휴리스틱에 가까워 5점까지는 어렵습니다.

**Productivity Impact & Problem Fit4/5**

명확한 개인 사용자와 실제 생산성 고충에는 잘 맞습니다. 퇴근 후 개인 아이디어를 머릿속에 붙잡아 두느라 작업기억이 잠식되는 문제를, 한 줄 캡처 → 즉시 착수 가능 여부 분류 → 정보 부족 항목의 사전조사 → 부하가 낮을 때 주 1회 한 건만 추천하는 흐름으로 풀어 생산성 개선 경로가 설득력 있습니다. 다만 절약 시간, 감소 단계, 오류 예방률 같은 정량 전후 비교는 없고, 라이브 스카우트도 실제 아이디어 제출·사전조사·결정 저장까지는 확인하지 않아 ‘명백히 큰 효과’까지 입증되지는 않았습니다. 따라서 ‘Convincing’ 수준의 4점이 적절합니다.

**Azure AI & Cloud Integration4/5**

Azure AI 계층을 실제 애플리케이션 흐름에 의미 있게 연결했고, 배포도 Azure Container Apps에서 동작합니다. `backend/app/judgment.py`와 `backend/app/research.py`가 `provider={"type":"azure"}` 및 `AZURE_OPENAI_*` 환경변수로 Azure OpenAI를 사용하며, 사전조사에는 `AzureOpenAI` 직접 호출 폴백도 있어 단순 호스팅이 아니라 AI 기능 자체가 Azure AI에 의존합니다. `infra/main.bicep`는 Container Apps, ACR, Log Analytics, 스케일 규칙, secretRef 기반 설정을 포함해 배포 자동화와 관측성/확장성의 일부를 갖췄습니다. 다만 Azure OpenAI 리소스 자체는 IaC로 만들지 않고 기존 리소스를 전제로 하며, Managed Identity나 Key Vault 연동 없이 API 키와 ACR admin/password secret을 사용하므로 5점의 클라우드 네이티브 보안 수준에는 미치지 못합니다.

**Functionality & Technical Execution4/5**

엔드투엔드는 통과했고, 핵심 API와 UI 흐름이 실제 배포에서 동작합니다. 백엔드는 FastAPI + SQLite로 PRD의 주요 계약(아이디어 캡처/판정, UserState, 사전조사, 주간 추천, 사람 결정)을 분리된 모델·DB·판정·리서치 계층으로 구현했고, AI 의존 실패 시 휴리스틱/직접 Azure 호출 폴백을 두어 캡처가 500으로 무너지지 않게 한 점이 좋습니다. 다만 PRD의 ‘주 1회 스케줄 배치’와 ‘덤프 5개 이상일 때 embedding similarity 랭킹’은 실제 구현이 수동 엔드포인트와 단순 우선순위에 머물러 있고, 테스트는 실제 Copilot SDK 경로를 비활성화한 휴리스틱 중심이라 5점 수준의 완성도까지는 아닙니다.

**User Experience & Workflow Design4/5**

전반적으로 매끄러운 사용자 경험입니다. PRD가 요구한 ‘한 줄 캡처 → AI 판정 → 추천/인박스에서 사람의 선택’ 흐름이 화면 구조와 문구에 잘 드러나고, 한국어 마이크로카피가 사용자의 부담을 낮춥니다. AI 대기 시간에는 진행 표시와 순환 문구, 스켈레톤을 제공하고, 추천 카드에는 3가지 근거와 수락/연기/제외 선택지가 있어 AI가 결정을 대체하지 않는다는 원칙도 잘 보입니다. 다만 핵심 입력창이 placeholder에 의존해 명시적 label이 부족하고, 사전조사 실패를 조용히 무시하는 부분, AI 작업 취소/결정 되돌리기/삭제 확인 같은 사용자 통제 장치가 제한적이어서 ‘매끄러움’ 수준으로 평가합니다.

**Responsible AI, Security & Trust3/5**

기본 수준으로 평가합니다. 하드코딩된 API 키는 확인되지 않았고, AI 산출물은 UI에서 “AI 제안”, “추천 근거”, “조사노트” 등으로 식별되며 추천 수락/연기/제외는 사람의 버튼 선택으로 남겨져 있습니다. 또한 Pydantic 입력 길이 제한, 구조화된 도구 호출, JSON 출력 요구 등 일부 환각 완화 장치가 있습니다. 다만 공개 배포된 단일 사용자 앱에 인증/권한 분리가 없어 누구나 아이디어 조회·삭제 API를 호출할 수 있는 구조이고, Key Vault/managed identity는 보이지 않으며, 프롬프트 인젝션 방어는 명시적이라기보다 도구 스키마에 의존합니다. 삭제 같은 파괴적 동작도 별도 확인 없이 실행됩니다.

**Innovation & Originality4/5**

AI를 단순 채팅·요약·문서검색에 붙인 형태가 아니라, 사용자의 작업기억을 보호한다는 뚜렷한 제품 관점에서 아이디어를 `inbox`/`dump`로 판정하고, 정보 부족 항목만 조용히 사전조사한 뒤 주 1개만 제안하는 흐름을 설계해 ‘Inventive’에 해당합니다. Copilot SDK 도구 호출로 정보 준비성·현재 여유를 분리 평가하고, 사전조사도 자료와 선택지로 제한해 인간의 최종 결정을 남기는 점은 창의적입니다. 다만 핵심 AI 기법 자체는 LLM 분류·리서치·랭킹의 조합이고, 코드상 주간 추천의 임베딩 연관성은 아직 고정 문구/단순 우선순위에 가까워 ‘Groundbreaking’까지는 아닙니다.
