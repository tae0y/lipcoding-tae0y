# FEEDBACK — Competition Judging Results

**Overall Assessment**

"Tteoolim (Working Memory Inbox)" demonstrates a clear problem awareness around protecting the user's working memory, and implements it as a persuasive product flow: single-line capture → AI judgment → pre-research for information-lacking ideas → low-load weekly recommendation. Rather than a simple chat wrapper, the app uses `CopilotClient.create_session` and tool calls to assess idea readiness and user capacity, assembling pre-research results as structured data — aligning the Copilot SDK's role tightly with the product's core value. The actual deployment on FastAPI, SQLite, Azure OpenAI, and Azure Container Apps is well-established, and the AI failure fallback adds stability. The UX is generally smooth: microcopy, progress indicators, skeletons, recommendation rationale, and accept/postpone/exclude options help users reduce cognitive load while retaining the final decision. The design principle of "quietly research only information-lacking ideas, then propose one thing when load is low" is a creative fit for the productivity problem. However, some core promises remain limited: weekly recommendation is closer to a manual endpoint and simple priority than a real scheduled batch or embedding-based ranking; tool calls are used primarily to structure model output rather than perform external searches. Productivity impact also lacks quantitative evidence. Additionally, the absence of auth in a public deployment, key-based secrets without Key Vault or managed identity, and limited user control (delete confirmation, undo, decision reversal) are significant weaknesses in trust and safety. The highest-impact improvement is to strengthen reliability and automation as a production-ready personal assistant: reinforce auth, secret management, and undo guards first, then connect the weekly recommendation to a real scheduler with embedding-based relevance scoring.

**Effective Use of Copilot SDK — 4/5**

Copilot SDK usage is strong, directly tied to the app's core value of idea triage and info_gap pre-research. Rather than a simple chat wrapper, it attaches tools to `CopilotClient.create_session` to judge information readiness and current capacity, assembling structured tool-call results into a final data model for pre-research with SSE streaming. However, tools are primarily used to structure model output rather than perform real external searches or data lookups, and the weekly recommendation's embedding similarity / advanced context management is still closer to heuristics — making a full 5 difficult.

**Productivity Impact & Problem Fit — 4/5**

Fits a clear individual user and a real productivity pain point. The flow — single-line capture → actionability classification → pre-research for information-lacking items → one recommendation per week when load is low — makes the productivity improvement path persuasive. However, there is no quantitative before/after comparison (saved time, reduced steps, error prevention rate), and the live scout did not confirm the full flow through idea submission → pre-research → decision save. "Convincing" at 4 is appropriate.

**Azure AI & Cloud Integration — 4/5**

Azure AI is meaningfully connected to the actual application flow, and deployment runs on Azure Container Apps. `backend/app/judgment.py` and `backend/app/research.py` use Azure OpenAI via `provider={"type":"azure"}` and `AZURE_OPENAI_*` environment variables, with a direct `AzureOpenAI` fallback for pre-research. `infra/main.bicep` includes Container Apps, ACR, Log Analytics, scale rules, and secretRef-based configuration. However, the Azure OpenAI resource itself was not created as IaC (presupposed as existing), and API keys plus ACR admin/password secrets are used without Managed Identity or Key Vault integration — falling short of 5-point cloud-native security.

**Functionality & Technical Execution — 4/5**

End-to-end passes; core APIs and UI work in the actual deployment. The backend implements the PRD's main contract (idea capture/judgment, UserState, pre-research, weekly recommendation, human decision) across separate model/DB/judgment/research layers in FastAPI + SQLite, with heuristic/direct Azure fallbacks preventing capture from crashing on AI failures. However, the PRD's "weekly scheduled batch" and "embedding similarity ranking when dump ≥ 5" remain a manual endpoint and simple priority ordering, and tests disable the actual Copilot SDK path — not quite 5-point completeness.

**User Experience & Workflow Design — 4/5**

Generally smooth. The PRD's required flow — single-line capture → AI judgment → recommendation/inbox with human choice — is clearly expressed in screen structure and copy. During AI wait time, progress indicators, cycling copy, and skeletons are provided; the recommendation card shows three rationale items plus accept/postpone/exclude options. However, the main input relies on placeholder text with no explicit label, pre-research failures are silently ignored, and user control mechanisms — AI task cancel / decision reversal / delete confirmation — are limited. Rated "smooth" at 4.

**Responsible AI, Security & Trust — 3/5**

Rated at a basic level. No hardcoded API keys were found; AI outputs are labeled in the UI as "AI suggestion," "recommendation rationale," "research notes"; recommendation accept/postpone/exclude is left as a human button choice. Some hallucination mitigation exists: Pydantic input length limits, structured tool calls, JSON output requirements. However, the publicly deployed single-user app has no auth/authorization separation (anyone can call idea read/delete APIs), Key Vault/managed identity is absent, and prompt injection defense relies on tool schema rather than being explicit. Destructive operations like deletion execute without confirmation.

**Innovation & Originality — 4/5**

Rather than attaching AI to simple chat, summarization, or document search, this app designs around a distinct product perspective — protecting working memory — classifying ideas as `inbox`/`dump`, quietly pre-researching only information-lacking items, and proposing one item per week. This qualifies as "Inventive." Using Copilot SDK tool calls to separately evaluate information readiness and current capacity, and limiting pre-research output to materials and options to preserve the human's final decision, is creative. However, the core AI techniques are a combination of LLM classification, research, and ranking, and weekly recommendation embedding relevance is still closer to fixed phrases/simple priority — not quite "Groundbreaking."
